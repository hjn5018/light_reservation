# 🚀 웹소켓(WebSocket) 도입 타당성 및 아키텍처 분석 보고서

본 문서는 현재 `light_reservation` 시스템의 기본 통신 방식인 **TCP Raw Socket** 구조를 **웹소켓(WebSocket)** 구조로 전환(Refactoring)할 때의 아키텍처 변화와 장단점, 그리고 기술적 고려사항을 분석한 설계 검토 문서입니다.

---

## 1. 아키텍처 변화 비교

### 1.1 현재 방식 (HTTP Polling + TCP Raw Socket)
현재 시스템은 웹 브라우저의 제약을 극복하기 위해 중간에 파이썬 CGI 게이트웨이를 두는 다단계 요청 구조를 취하고 있습니다.
* **데이터 흐름**: `웹 브라우저 (JS)` ➔ *(HTTP Fetch)* ➔ `Pi A (Apache / CGI)` ➔ *(TCP 소켓 포트 50007)* ➔ `Pi B (Socket Server Thread)`
* **상태 동기화**: 최신 상태를 유지하기 위해 프론트엔드가 5초 주기로 서버에 계속 자원을 요청(Polling)합니다.

### 1.2 변경 방식 (전이중 웹소켓 단일 파이프라인)
웹소켓(예: `Flask-SocketIO`)을 도입하면 브라우저 자바스크립트가 하드웨어 서버와 직접 상시 연결을 유지합니다.
* **데이터 흐름**: `웹 브라우저 (JS)` ⇄ *(Websocket 통신)* ⇄ `Pi B (Flask-SocketIO 서버)`
* **상태 동기화**: 서버에서 상태가 변경되는 즉시 연결된 모든 클라이언트에 데이터를 밀어주는 실시간 푸시(Real-time Push) 방식으로 전환됩니다.
* **Pi A의 역할 축소**: 중간 중계용 CGI(`update_status.py`)가 불필요해지며, Pi A는 정적 웹 리소스(`index.html`, `app.js`)만 배포하는 순수 웹서버 역할만 수행합니다.

---

## 2. 웹소켓 도입의 주요 장점

1. **진정한 실시간 피드백 (Real-time Linkage)**
   * 타 사용자가 예약을 선점하거나, 백그라운드 스케줄러 데몬 스레드에 의해 예약/사용 시간이 자동 만료되는 순간 즉시 화면이 갱신됩니다. 최대 5초간 존재하던 화면 지연 현상이 0.1초 미만으로 단축됩니다.
2. **네트워크 및 시스템 부하 감소**
   * 5초마다 주기적으로 HTTP Connection을 맺고 끊으며 대량의 HTTP 헤더를 주고받던 오버헤드가 사라집니다. 하나의 가벼운 웹소켓 커넥션을 유지하므로 라즈베리 파이의 네트워크 자원을 아낄 수 있습니다.
3. **정교한 타이머 동기화**
   * 현재 프론트엔드(`app.js`) 카운트다운 타이머와 백엔드 스케줄러 간의 미세한 시간 차이를 보정하기 위한 폴링 로직이 생략되고, 백엔드가 만료 타이밍을 완벽하게 제어하여 브라우저에 신호를 보낼 수 있습니다.

---

## 3. 전환 시 고려사항 및 단점 (Trade-off)

1. **라즈베리 파이 메모리 커넥션 부담**
   * TCP Raw Socket은 명령을 처리한 후 즉시 소켓을 닫아 리소스를 반환하지만, 웹소켓은 사용자가 브라우저를 켜놓은 동안 계속 연결을 유지(Persistent Connection)해야 합니다. 동시 접속자가 많아질 경우 라즈베리 파이 B의 자원 사용량을 모니터링해야 합니다.
2. **의존성 라이브러리 증가**
   * 파이썬 내장 `socket` 모듈 외에 백엔드 구동을 위한 `Flask-SocketIO` 및 비동기 네트워킹 라이브러리(`gevent` 또는 `eventlet`)의 추가 관리가 필요합니다.
3. **CORS 및 보안 설정 관리**
   * 브라우저 자바스크립트가 외부 장치(Pi B)의 웹소켓 포트로 직접 커넥션을 열어야 하므로, 웹소켓 통신에 대한 CORS(Cross-Origin Resource Sharing) 규칙을 명확하게 허용해 주어야 브라우저 단에서 차단되지 않습니다.
